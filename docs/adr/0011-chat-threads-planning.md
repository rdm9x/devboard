## TL;DR (для не-программистов)

Чат с Управляющим становится **главным интерфейсом** общения, переехав из узкой правой панели в полноценную страницу `/chat`. Layout по образцу Claude.ai: слева список диалогов (топики), центр — выбранный диалог, справа — панель «Собрать планёрку» + список текущих задач Управляющего в реальном времени.

**Планёрка** — новый паттерн взаимодействия: ты создаёшь новый диалог, выбираешь каких лидов отделов пригласить, пишешь проблему/задачу. Лиды **обсуждают между собой** в этом thread (спорят, выдвигают подходы). В конце Управляющий формирует итоговый отчёт с шагами решения. Ты апрувишь → Управляющий декомпозирует в задачи отделов.

## 1. Context

### 1.1. Текущая боль (owner-feedback 2026-05-25)

Owner показал пример типичного сообщения от Управляющего:

> *«Управляющий — сессия 8 (2026-05-25). Состояние на старт. Чат: 100 сообщений. Последнее от тимлид'а (id=137): ADR-009 Phase 2 эпик декомпозирован, B1 в review. ... Safety-net блокировка на B1+A1+A2+B2+F1+Q1 указывает что эти задачи требуют специальной обработки... Жду указаний 🙏»*

20+ строк формального отчёта, технический жаргон («safety-net блокировка», «B1+A1+A2+B2+F1+Q1», «вендорские плагины»), ноль действий («Рекомендация: спросить тимлида»), некорректная интерпретация ситуации (Управляющий думает что safety-net блокирует, хотя задачи реально сделаны).

### 1.2. Что ломает UX

1. **Узкая правая панель** — длинные сообщения нечитаемы.
2. **Не интерактивно** — owner пишет → ждёт ручного запуска сессии → ответ приходит «одним блоком».
3. **Шум** — в одном потоке смешаны owner-сообщения, отчёты тимлида, safety-net warnings.
4. **Нет действий** — упомянутые задачи нельзя принять/открыть из чата.
5. **Планёрка не работает** — ADR-009 §2.4 спроектировал паттерн, но UI не сделан, реальные обсуждения между лидами не происходят.

### 1.3. Что хочет owner (видение 2026-05-25)

> *«Отдельная вкладка для общения. Принцип как в чате Клауд: слева топики, потом чат это 70% экрана, правее 2 блока вертикально. Верхний — Собрать планёрку с выбором отделов. Нижний — задачи Управляющего в реалтайме.»*
>
> *«Планёрка это обсуждение для проработки задачи, выработки лучшего решения путём спора и обсуждения разными руководителями отделов. После планёрки — отчёт в чат.»*

## 2. Decision

### 2.1. Layout — новая страница `/chat`

Три колонки:

```
┌──────────────┬─────────────────────────────────┬──────────────────────┐
│  THREADS     │  ACTIVE THREAD                   │  📋 СОБРАТЬ ПЛАНЁРКУ  │
│  (~20%)      │  (~50%)                          │  (~30%, top 30%)     │
│              │                                  │                      │
│  📌 General  │  ┌─ Управляющий, 18:18 ────────┐│  Пригласить отделы:  │
│  💬 Лендинг  │  │ Тимлид доделал rename и     ││  ☑ Маркетинг        │
│  ⚙️ Деплой   │  │ workspace. 6/17 закрыто.    ││  ☐ Разработка       │
│  ...         │  └──────────────────────────────┘│  ☑ Юристы           │
│              │                                  │  [Создать планёрку] │
│  + Новый     │  ┌─ Ты, 18:20 ─────────────────┐│                      │
│              │  │ Подскажи тимлиду пофиксить  ││  ────────────────    │
│              │  └──────────────────────────────┘│                      │
│              │                                  │  📌 ЗАДАЧИ          │
│              │  [Ввод сообщения...]             │  УПРАВЛЯЮЩЕГО       │
│              │  [Отправить] [Запустить сессию]  │  (live, bottom 70%) │
│              │                                  │                      │
│              │                                  │  • Принять #abc12   │
│              │                                  │  • Запустить #def34 │
│              │                                  │  • Спросить marketing│
│              │                                  │                      │
│              │                                  │  + Добавить задачу  │
└──────────────┴─────────────────────────────────┴──────────────────────┘
```

### 2.2. Типы threads

| kind | Кто участвует | Назначение |
|---|---|---|
| `direct` | Owner ↔ Управляющий | Дефолтный «📌 General» — повседневные вопросы. Создаётся при первом запуске автоматически. |
| `planning` | Owner + Управляющий + N лидов | Планёрка по проблеме. Лиды обсуждают между собой, формируют отчёт. |

### 2.3. БД схема

```sql
CREATE TABLE chat_threads (
  id            TEXT PRIMARY KEY,                              -- uuid
  title         TEXT NOT NULL,                                 -- "Лендинг outdoor billboards"
  kind          TEXT NOT NULL CHECK (kind IN ('direct','planning')),
  participants  TEXT NOT NULL DEFAULT '[]',                    -- JSON: [{role_slug, joined_at}]
  status        TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active','finished','archived')),
  created_at    INTEGER NOT NULL,
  updated_at    INTEGER NOT NULL,
  finished_at   INTEGER,
  -- planning-specific (NULL для direct)
  source_problem            TEXT,                              -- что owner написал в начале
  decision_summary          TEXT,                              -- итоговый отчёт Управляющего
  decision_approved_at      INTEGER,                           -- когда owner апрувнул
  decision_created_tasks    TEXT                               -- JSON: [{task_id, dept_id}]
);

ALTER TABLE chat_messages ADD COLUMN thread_id TEXT REFERENCES chat_threads(id);
CREATE INDEX idx_chat_thread ON chat_messages(thread_id, created_at);

-- Дефолтный thread создаётся в setup.py / при первом старте:
INSERT INTO chat_threads (id, title, kind, participants, status, created_at, updated_at)
VALUES ('default', '📌 General', 'direct',
        '[{"role_slug":"owner"},{"role_slug":"managing-director"}]',
        'active', strftime('%s','now'), strftime('%s','now'));
```

### 2.4. Planning flow (полная схема)

```
   Owner                  Backend             Управляющий      Лиды (N)
   ─────                  ───────             ───────────      ───────
   1. Клик «+Планёрка»
   2. Выбирает отделы ─→ создаёт thread
   3. Пишет проблему    (planning, status='active')
                        ↓ params: thread_id, participants, problem
                        
                        ┌─ Phase 1: Опрос ──────────────────────────┐
                        │ Запускает 1 сессию Управляющего:          │
                        │   → читает problem, контекст              │
                        │   → пишет в thread:                       │
                        │     "Тема: <X>. Лиды, ваши подходы?"  ─┐ │
                        │                                       │  │
                        │ Запускает N сессий лидов параллельно: │  │
                        │   каждый: → читает thread + problem ──┘  │
                        │           → формирует свой подход        │
                        │           → постит в thread              │
                        └───────────────────────────────────────────┘
                        
                        ┌─ Phase 2: Обсуждение (2-3 раунда) ────────┐
                        │ Для каждого раунда:                      │
                        │   Управляющий перезапускает лидов:       │
                        │     → читают всё что написано в thread   │
                        │     → реагируют на чужие подходы         │
                        │     → постят новые идеи / возражения     │
                        │   Лимит: max 3 раунда                    │
                        └───────────────────────────────────────────┘
                        
                        ┌─ Phase 3: Синтез ─────────────────────────┐
                        │ Управляющий:                              │
                        │   → читает весь thread                    │
                        │   → пишет финальный отчёт:                │
                        │     "Решение: <предлагаемые шаги>"        │
                        │   → сохраняет в thread.decision_summary   │
                        │   → нотифицирует owner: "планёрка готова" │
                        └───────────────────────────────────────────┘
   4. Читает отчёт
   5. [Принять] / [Отклонить] / [Доработать]
   6. Если accept ─────→ создаёт N cross-task            
                        для отделов на основе decision   ─→ Управляющий
                                                            будет
                                                            распределять
                                                            в задачи
```

### 2.5. Realtime tasks block (правый нижний)

Источник: `list_tasks(assignee='managing-director')` + `WHERE status IN ('todo','wip','needs_approval')`.

Auto-refresh:
- SSE endpoint `/api/manager/tasks/stream` (по образцу `/api/team/stream`).
- Каждое изменение → push в браузер.
- Альтернатива (проще для MVP): poll каждые 5 сек.

Кнопка «+ Добавить задачу» в этом блоке — открывает modal для создания задачи **сразу на Управляющего**.

### 2.6. Navigation

- Sidebar: пункт «💬 Чат» — теперь главный.
- Правая chat-панель **удаляется** с главной (Kanban-страница `/`).
- Default thread (`📌 General`) автоматически создаётся при первом запуске setup.
- При клике «+ Новый диалог» owner выбирает тип:
  - **Прямой с Управляющим** (1:1)
  - **Планёрка** (с приглашением отделов)

### 2.7. Communication patterns

| Раньше | Теперь |
|---|---|
| Owner пишет в правую панель → ждёт session → большой отчёт | Owner пишет в active thread → быстрый ответ Управляющего (короткий) |
| Управляющий пишет 20-строчные отчёты | Управляющий: ≤6 строк на ответ. Длинное — в attached document |
| Шумно (отчёты тимлида, safety-net) | Тимлид-отчёты идут в `📌 Лог` thread (kind='direct', participants=[managing,owner]) или отдельную вкладку |
| Действия — иди в Inbox | Inline buttons: `[Принять #X]`, `[Открыть]`, `[Добавить в задачи]` |

## 4. Consequences

### Плюсы

- **Главный интерфейс** — owner общается через современный chat-UI как привык в ChatGPT/Claude.
- **Threads = история** — каждая большая задача имеет свой thread с обсуждением.
- **Планёрка работает** — multi-agent dialog реализуется впервые в проекте, паттерн ADR-009 §2.4 становится живым.
- **Чистый поток** — owner ↔ Управляющий не загромождён отчётами/warnings.
- **Realtime видимость** — owner видит что Управляющий «делает прямо сейчас», как в Linear.

### Минусы / риски

- **Стоимость планёрки**: 4 лида × 3 раунда × sonnet/opus = ~$2-5 за планёрку. Нужны лимиты (Phase 3b B4).
- **Orchestration сложна**: запуск N сессий параллельно + read shared thread state + round-robin synchronisation. Risk: race conditions, dead-lock.
- **UX переезд**: правая chat-панель убирается с главной — owner может потерять привычку. Mitigation: при первом запуске `/chat` показать onboarding-overlay.
- **Главная остаётся Kanban**: `/chat` отдельная страница. Owner должен переключаться (Sidebar). Альтернатива — сделать `/chat` главной, но это переломно для текущего UX. Обсудим в Open Questions.

## 5. Alternatives Considered

### 5.1. Улучшить sidebar-чат без переноса

Расширить ширину sidebar, добавить inline-кнопки. **Отвергнуто** owner-ом: «нужна отдельная вкладка».

### 5.2. Один глобальный thread без topics

Все сообщения в одной хронологии (как сейчас). **Отвергнуто**: теряется история обсуждений отдельных проблем.

### 5.3. Каждой задаче — свой thread

Auto-create thread для каждой `tasks` записи. **Отвергнуто**: 269 done = 269 threads, шум. Threads только для значимых обсуждений (планёрок) + дефолтный direct.

### 5.4. Slack/Discord integration вместо своего

Использовать Slack как фронт. **Отвергнуто**: Devboard — локальный продукт, не должен зависеть от внешнего сервиса. Также см. ADR-008 (channels — deferred).

### 5.5. Realtime через WebSocket вместо SSE

WebSocket даёт двустороннюю связь. **Не нужно**: chat owner → server идёт через REST POST, server → owner через SSE — этого достаточно.

## 6. Resolved Decisions (owner 2026-05-25)

1. **Главная страница — Kanban остаётся главной** (`/`). `/chat` — отдельная вкладка (sidebar пункт «💬 Чат»).
2. **Тимлид-отчёты не показываются owner-у вообще** — owner видит в чате только свои сообщения и сообщения Управляющего. Сообщения тимлида/лидов/специалистов фильтруются на уровне endpoint. Управляющий продолжает их видеть и читать (его контекст).
3. **Title планёрки** — owner вводит сам при создании thread. Не LLM-генерация.
4. **Стоп-сигнал планёрки** — кнопка «Остановить» в активном thread с типом `planning`. Завершает thread со статусом `aborted`, kills запущенные lead-сессии.
5. **Архив** — отдельная секция «📁 Архив» в Threads list (под основным списком active threads). Завершённые/aborted threads автоматически туда.
6. **Раунды планёрки** — owner выбирает перед запуском от **1 до 5** (default 3). UI control в правой панели «Собрать планёрку».
7. **Лимит участников** — max 4 лида + Управляющий. Hardcoded — обоснование: после 4 параллельных голосов обсуждение становится шумным.
8. **Нотификации** — отложено до ADR-008 channels.

### 6.1. Что меняется в реализации после resolve

| Решение | Технический impact |
|---|---|
| 6.2 (фильтр тимлид-сообщений) | Endpoint `GET /api/threads/<id>/messages?viewer=owner` фильтрует `WHERE author NOT IN (lead-slugs)`. Для `viewer=managing-director` — без фильтра. |
| 6.3 (owner вводит title) | Modal «+ Новый диалог» имеет input для title. Required. |
| 6.4 (Остановить) | `POST /api/threads/<id>/stop` → `status='aborted'`, kill lead subprocesses. |
| 6.5 (Архив) | Threads list имеет 2 секции: `[Active]` + `[Архив (collapsible)]`. |
| 6.6 (раунды 1-5) | В UI «Собрать планёрку» dropdown «Раундов: 1/2/3/4/5» (default 3). Backend orchestrator принимает `rounds` параметр. |

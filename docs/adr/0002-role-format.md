# ADR-002 — Формат файлов ролей `роли/*.md`

- **Status:** Proposed (2026-05-21)
- **Date:** 2026-05-21
- **Authors:** архитектор (devboard)
- **Epic:** E7 — Формализация ролей (parent task `daa44bdea2f7`)
- **Supersedes:** —
- **Related:** ADR-001 (`docs/adr/0001-llm-provider.md`) — frontmatter роли используется фабрикой `create_provider`.

## 1. Context

Сейчас формат файлов в `роли/*.md` **implicit**. Существующие 7 ролей (`тимлид`, `архитектор`, `бэкенд`, `qa`, `devops`, `frontend`, `техписатель`) написаны от руки и используют свой произвольный YAML-frontmatter:

```yaml
---
тип: системный_промт_роли
роль: бэкенд
проект: devboard
дата_создания: 2026-05-20
описание_короткое: |
  Системный промт для subagent'а Claude когда он играет роль …
---
```

Загрузчик ролей — простой `cat роли/<role>.md` в `команды/devboard-work.sh` (см. строку 25). Никакой схемы, никакой валидации, никакого контракта с кодом — поля frontmatter не читаются программно вообще.

Это работает пока ролей семь и их пишет один человек. Проблемы возникают **в трёх местах одновременно**:

1. **E6 — Multi-LLM (ADR-001).** Чтобы фабрика `create_provider` смогла выбрать движок per-role (`llm: claude|openai|ollama`), у файла роли должны быть машино-читаемые поля `llm`, `model`, опционально `temperature`/`max_tokens`. Сейчас их негде взять.
2. **Open-source readiness.** Пользователь с GitHub склонирует репо и захочет добавить свою роль (например `data-scientist.md`). Без явной спецификации он не знает: какие поля обязательны, какие опциональны, что будет если поле опечатать, как ограничить роли в MCP-tools.
3. **MCP tools per role.** Тимлид и архитектор имеют доступ ко всем MCP-tools, но `qa` не нужен `create_task`, а `frontend` не нужен `propose_candidate`. Сейчас разграничения нет — все 25+ MCP-инструментов доступны каждой роли. Хотим декларативный allowlist.

Дополнительно: длина `тимлид.md` уже **23 KB** — близка к разумному пределу одного системного промта. Без max-size constraint следующая роль может разрастись до 100 KB и сжечь контекст-окно ещё до первого запроса пользователя.

Цель этой ADR — формализовать формат файлов ролей: какие поля frontmatter обязательны, какие опциональны, какой у них тип, как валидируется, как версионируется.

## 2. Decision

### 2.1. Структура файла

Файл роли = `<YAML-frontmatter>` + `<markdown-тело>`. Разделитель — `---` в первой строке и `---` ниже. Тело файла после второго `---` — это **системный промт** роли, передаётся как `system_prompt` в `LLMProvider.invoke` (см. ADR-001 §2.1).

Итоговый каноничный пример:

```yaml
---
schema_version: 1
name: backend
description: Python backend dev — writes code, tests, configs.
llm: claude
model: claude-sonnet-4-5
tools:
  - mcp__pride-tasks__get_task
  - mcp__pride-tasks__add_comment
  - mcp__pride-tasks__submit_result
  - mcp__pride-tasks__claim_task
  - Read
  - Write
  - Edit
  - Bash
temperature: 0.3
max_tokens: 8192
---
# You are a backend developer of devboard
…systemprompt…
```

### 2.2. Обязательные поля

| Поле | Тип | Ограничения | Зачем |
|---|---|---|---|
| `schema_version` | `int` | `== 1` | Версия формата. При breaking-изменениях формата инкрементируем; старые роли продолжают грузиться через адаптер (см. §2.6). |
| `name` | `str` | slug `^[a-z][a-z0-9-]{1,31}$`, уникален в `роли/` | Машинный идентификатор роли. Тимлид вызывает по `name`, а не по имени файла. |
| `description` | `str` | 1…100 символов, single-line | Краткое описание для дашборда/CLI-листинга. Не системный промт. |
| `llm` | `str` | `claude` \| `openai` \| `ollama` | Какой провайдер использовать (см. ADR-001 §2.5). |
| `model` | `str` | непустая строка | Конкретная модель провайдера. Валидность ID — на стороне провайдера (мы не хардкодим список моделей, чтобы новые релизы не требовали правки кода). |

Отсутствие любого из этих 5 полей → `RoleConfigError` при загрузке. Это сознательный strict-mode: лучше упасть на старте, чем выдать subagent'у пустой системный промт.

### 2.3. Опциональные поля

| Поле | Тип | Default | Назначение |
|---|---|---|---|
| `tools` | `list[str]` \| `"*"` | `"*"` (все доступные инструменты) | Allowlist MCP-инструментов и встроенных tools (`Read`, `Write`, `Bash`, …). `"*"` = все. Пустой список `[]` = роль без инструментов (только текст). |
| `temperature` | `float` | `0.3` | Температура сэмплинга. `0.0…2.0`. У Ollama-моделей маппится в `options.temperature`. |
| `max_tokens` | `int` | `8192` | Максимум output-токенов. `1…200000`. Для Anthropic кладётся в `max_tokens`, для OpenAI — в `max_completion_tokens`, для Ollama — в `options.num_predict`. |
| `extras` | `dict` | `{}` | Провайдер-специфичные параметры, которые не вписываются в общий контракт. Пример: `{cli_flags: ["--permission-mode", "bypassPermissions"]}` для `ClaudeCLIProvider`. Игнорируется другими провайдерами. |

Прочие поля (`тип`, `проект`, `дата_создания`, `описание_короткое`, `роль`) из существующих файлов — **legacy**, попадают в `extra_metadata: dict` без валидации, не используются кодом. См. §2.6 о миграции.

### 2.4. Размер файла

**Hard limit: 50 KB** на файл роли. При превышении — `RoleConfigError("Role file too large: {n} bytes > 50KB")`.

Обоснование: системный промт целиком уходит в контекст-окно на **каждом** запросе subagent'а. 50 KB ≈ 12.5K токенов — это уже 6% от 200K-окна Claude и 16% от 80K-окна gpt-4o-mini. Файл крупнее = роль начнёт жрать контекст до того, как пользователь сказал хоть слово. Текущий максимум — `тимлид.md` (23 KB) — комфортно вписывается с запасом 2x.

Это **soft architectural limit**, не технический. Если когда-то понадобится больше — поднимаем в новой ADR-002-rev с явным обоснованием. YAGNI.

### 2.5. Валидация — pydantic.BaseModel

Реализация (псевдо-код, детали — задача бэкенду):

```python
from pydantic import BaseModel, Field, field_validator
from typing import Literal

class RoleConfig(BaseModel):
    """Конфигурация роли из frontmatter файла роли/<name>.md."""

    schema_version: Literal[1]
    name: str = Field(pattern=r"^[a-z][a-z0-9-]{1,31}$")
    description: str = Field(min_length=1, max_length=100)
    llm: Literal["claude", "openai", "ollama"]
    model: str = Field(min_length=1)

    tools: list[str] | Literal["*"] = "*"
    temperature: float = Field(default=0.3, ge=0.0, le=2.0)
    max_tokens: int = Field(default=8192, ge=1, le=200_000)
    extras: dict = Field(default_factory=dict)

    # legacy + неизвестные поля — собираем для совместимости
    model_config = {"extra": "allow"}

    @field_validator("tools")
    @classmethod
    def _validate_tools(cls, v):
        if v == "*":
            return v
        if not isinstance(v, list) or not all(isinstance(t, str) for t in v):
            raise ValueError("tools must be '*' or list[str]")
        return v


def load_role(path: Path) -> tuple[RoleConfig, str]:
    """Парсит файл роли. Возвращает (config, system_prompt_body).

    Бросает RoleConfigError при любых проблемах с frontmatter или размером.
    """
    if path.stat().st_size > 50 * 1024:
        raise RoleConfigError(f"Role file too large: {path.stat().st_size} bytes")
    frontmatter, body = _split_frontmatter(path.read_text(encoding="utf-8"))
    try:
        config = RoleConfig.model_validate(yaml.safe_load(frontmatter))
    except ValidationError as e:
        raise RoleConfigError(f"Invalid frontmatter in {path}: {e}") from e
    return config, body
```

`RoleConfigError` — отдельный exception-класс, дашборд и `devboard-work.sh` ловят его и показывают пользователю человеко-читаемое сообщение «роль `xxx.md` не загружена: …», но **не падают целиком** — продолжают работать с остальными ролями.

### 2.6. Совместимость с существующими ролями

Текущие 7 файлов в `роли/` **не валидны** под новый формат (нет `schema_version`, `name`, `llm`, `model`; зато есть legacy-поля `тип`, `роль`, `проект`, `дата_создания`, `описание_короткое`).

Стратегия миграции — **одноразовый PR от бэкенда**:

1. Бэкенд (задача-наследник E7.2) добавляет к каждому из 7 файлов новые поля frontmatter с правильными значениями. Legacy-поля **оставляет нетронутыми** (они валидны благодаря `extra=allow`), это сохраняет git-историю и читаемость.
2. Соответствие `name` ↔ файл фиксируется один раз: `тимлид.md` → `name: teamlead`, `архитектор.md` → `name: architect`, и т.д. С этого момента код обращается к роли по `name`, имя файла — деталь хранения.
3. `llm: claude` + `model: claude-sonnet-4-5` ставится дефолтом для всех существующих ролей (текущее поведение). При желании пользователь меняет per-role.
4. `tools` для каждой роли заполняется по таблице доступа — её составляет тимлид отдельной задачей; до этого момента `tools: "*"` (= текущее «всё доступно»). Сужение allowlist'а — некритично, отложенная работа, не часть E7.

Migration-PR не ломает обратную совместимость: до него `devboard-work.sh` продолжает делать `cat роли/тимлид.md` (всё работает), после него — `devboard-work.sh` начинает использовать новый загрузчик (задача E7.4, devops).

### 2.7. Версионирование схемы

`schema_version: 1` — текущая версия формата. Правила:

- **Добавление нового опционального поля** → не меняет версию (backward-compatible).
- **Добавление нового обязательного поля** → инкремент версии (`1 → 2`).
- **Удаление поля или смена семантики существующего** → инкремент версии.

Загрузчик поддерживает **N и N-1** одновременно (адаптер). N-2 — `RoleConfigError("schema_version 1 is no longer supported; please migrate to v3")`. Это даёт open-source-пользователям окно на миграцию между релизами.

## 3. Consequences

**Плюсы**

- **Multi-LLM работает per-role.** ADR-001 §2.5 фабрика `create_provider(config)` получает валидированный `RoleConfig` — никаких `KeyError`, никаких `if "llm" in config`.
- **Декларативное разграничение tools.** Можно дать `qa.md` allowlist без `create_task` и без `Write`, а архитектору — без `Bash`. Контракт читается из файла, а не зашит в кодовую базу.
- **Strict-mode на старте.** Опечатка в `model: claude-sonnnet-4-5` ловится при загрузке роли, а не на середине первого запроса subagent'а с непонятной 400-ошибкой от API.
- **Open-source onboarding.** Контрибьютор копирует `qa.md`, меняет `name`, `description` и системный промт — и сразу получает рабочую роль. Минимум магии.
- **Размер ограничен.** Не получится случайно засунуть в роль 500-страничную инструкцию и сжечь контекст до первого ответа.
- **Версия схемы фиксирует контракт.** Между минорными релизами devboard роли можно менять с гарантией обратной совместимости.

**Минусы / риски**

- **Migration cost.** 7 существующих файлов нужно обновить ровно одним PR. Если этот PR разъезжается во времени с релизом загрузчика — половина ролей сломается. Решение: миграция файлов и релиз парсера ролей идут в одной PR (задача E7.2).
- **Дубликат `name` ↔ filename.** Имя роли теперь есть и в имени файла (`архитектор.md`) и во frontmatter (`name: architect`). Если они разъедутся — путаница. Решение: загрузчик пишет warning в лог при несовпадении `stem(file) != name` (с учётом транслитерации), но не падает.
- **YAML-frontmatter с кириллицей.** Существующие файлы используют русские ключи (`тип`, `роль`). Новый формат — только latin keys. Это формальный breaking, хотя legacy-поля сохраняются как `extra=allow`. Контрибьютор увидит несоответствие и **может** сначала не понять, что в новом формате ключи latin. Mitigation: пример в README и в `docs/adr/0002-role-format.md` (этот документ).
- **`tools` как allowlist даёт ложное ощущение безопасности.** Это **не sandbox** — это фильтр на стороне нашего агент-loop'а. Если subagent уговорит другую роль вызвать заблокированный tool за него — никакая декларация в frontmatter этого не остановит. Это нормально для коллабораторного режима, но в публичных доках важно явно написать «`tools` — это разграничение, не security boundary».
- **50 KB limit когда-нибудь упрётся.** В долгосрочной перспективе тимлид (уже 23 KB) может попросить расширения. Решение — рефакторинг роли (вынос `## Алгоритм` в отдельный документ, на который ссылка), а не повышение лимита.

## 4. Alternatives Considered

### 4.1. JSON Schema вместо pydantic

**Отвергнуто.** Нам нужна валидация **внутри Python-кода** с runtime-объектом-конфигом, а не отдельный CLI-валидатор. Pydantic даёт и runtime-валидацию, и автогенерацию JSON Schema (`RoleConfig.model_json_schema()`) когда понадобится для документации/IDE-completion. Зависимость `pydantic>=2` уже в стеке (используется в MCP-сервере).

### 4.2. TOML вместо YAML

TOML — строже, меньше footguns (без неявных булевых `yes`/`no`, без значащих отступов). Но **YAML frontmatter — индустриальный стандарт** для Markdown-документов (Jekyll, Hugo, MkDocs, Obsidian, Pandoc, Anthropic skills, Claude Code skills). Любой инструмент, который смотрит на `*.md`-файл, ожидает YAML. Менять конвенцию ради двух процентов строгости — не окупается. Mitigation существующих YAML-проблем: `yaml.safe_load` (не `yaml.load`), pydantic-валидация всех типов на втором проходе.

### 4.3. JSON-файлы рядом (`роли/архитектор.md` + `роли/архитектор.json`)

Разделить системный промт (md) и конфиг (json). **Отвергнуто.** Лишний файл на каждую роль = легко рассинхронизировать (промт обновили, конфиг забыли). Frontmatter гарантирует один-файл-одна-роль.

### 4.4. Положить всю конфигурацию ролей в один `roles.yaml`

```yaml
roles:
  backend:
    description: ...
    llm: claude
    system_prompt: |
      Ты — бэкенд-разработчик...
```

**Отвергнуто.** Один файл на 7 ролей = git-merge-конфликты при параллельной работе нескольких контрибьюторов. И теряем фишку «открыл `роли/qa.md` в любом редакторе — увидел системный промт как нормальный markdown с подсветкой и оглавлением».

### 4.5. Не валидировать, парсить best-effort

«Если поле есть — используем, нет — дефолт». Текущее имплицитное поведение. **Отвергнуто** — именно это пытаемся починить (см. §1, пункт 2). Без strict-mode опечатка молча даёт неверное поведение в проде.

## 6. Open questions

### 6.1. Что делать с legacy-полями при сохранении файла?

Если кто-то открывает `роли/qa.md` через CLI-команду «отредактируй роль» (если она когда-нибудь появится), и при сохранении мы сериализуем `RoleConfig` обратно в YAML — теряем ли мы legacy-поля (`тип`, `проект`, `дата_создания`)?

**Текущее предложение:** при сохранении пишем сначала валидированные поля (`schema_version`, `name`, …), потом дописываем все ключи из `model_extra` без изменений. Это сохраняет читаемость и git-историю. Финальное решение — при появлении CLI «отредактируй роль», сейчас писатель файлов = только человек.

### 6.2. Stand-alone tools или groups?

Поле `tools` — плоский список. В будущем может захотеться **групп**: `tools: ["pride-tasks:*", "fs:read-only"]`. Это удобнее, но требует справочника групп.

**Текущее предложение:** не делаем сейчас. Если в `tools` встретилось `"*"` — все доступные. Если конкретный список — конкретный. Group-синтаксис — отдельная ADR-002-rev когда наберётся 3+ роли с длинными tools-списками, у которых явно видны повторы.

### 6.3. Кросс-roles inheritance?

«Все QA-роли начинаются с того же промта». Можно ввести `extends: base-qa.md`. **Текущее предложение** — не вводить. YAGNI. У нас 7 ролей, повторов нет. Если когда-нибудь будет — отдельная ADR.

# Approval-gates — как Дмитрий контролирует критичные операции

## Базовый принцип

Subagent (бэкенд, QA, или сам тимлид) **никогда не выполняет** критичную операцию напрямую. Вместо этого:
1. Создаёт под-задачу со статусом `needs_approval`.
2. В описании пишет ЧТО собирается сделать, ПОЧЕМУ, и какой command будет выполнен.
3. Ждёт пока Дмитрий не нажмёт Approve или Reject в дашборде.
4. После approve — выполняет операцию.
5. После reject — задача закрывается с пометкой «отказано», subagent уведомлён.

## Список критичных операций (требуют approval)

| Операция | Пример | Зона риска |
|---|---|---|
| `git push origin main` | Отправка кода большой системы на GitLab | Боевой код — нужен ревью |
| `git push --force` | Любой force-push | Удаление истории |
| `ssh root@<vps>` или `ssh <user>@prod-host` любые изменения | Перезапуск сервиса, миграция БД | Прод-инфра |
| `systemctl restart <service>` | Останов работающего сервиса | Прерывание пользователей |
| Создание/удаление/изменение прав user'ов в Bitrix24 | Изменение состава ботов | Бизнес-операция |
| Удаление файлов вне `/D.AI/команда/data/` | `rm -rf` на боевые директории | Безопасность |
| Удаление таблиц БД, `DROP TABLE` | Потеря данных | Данные |
| Изменение API-ключей, secret'ов | Ротация токенов | Security |
| Создание новых OAuth client'ов | Расширение доступа | Security |
| `pip install` глобально или с правами root | Изменение системного Python | Стабильность |
| Деплой новых ботов в Bitrix24 | Расширение производственного отдела | Бизнес-стратегия |

**Что НЕ требует approval (всё остальное):**
- Чтение любых файлов.
- Write/Edit файлов в `/D.AI/команда/` и подпроектах большой системы.
- Запуск тестов (pytest и т.п.).
- SQLite read/write в локальной БД канбана.
- Создание/обновление задач в канбане.
- Запросы к Bitrix24 REST API на чтение.
- `git commit` (без push).
- Локальная установка зависимостей в venv проекта.

## UI flow в дашборде

### Сторона subagent'а

Когда subagent определяет что нужна критичная операция (например, git push), он делает MCP-вызов:

```python
mcp_pride_tasks.create_task(
    title="git push origin main для коммита fd9ecdb",
    description="Загрузить новый код большой системы на GitLab. "
                "Коммит содержит: webhook-приёмник + bitrix24_client.py.\n\n"
                "Команда: git push origin main\n"
                "Файлы: pride_dev/webhook.py, pride_dev/bitrix.py\n"
                "Тесты: 14/14 зелёные",
    assignee="тимлид",           # или "дмитрий" если эскалация
    parent_id=<id текущей задачи>,
    status="needs_approval",      # ключевое
    labels=["approval", "git-push"],
)
```

После этого subagent **возвращает результат** в свою задачу с пометкой «жду approval-задачу #N» и **завершает работу**.

Тимлид при следующем проходе видит, что у его задачи под-задача в needs_approval — он сам тоже не двигается дальше, пока Дмитрий не решит.

### Сторона Дмитрия

В дашборде колонка **NEEDS APPROVAL ⚠** с бэйджем количества. Дмитрий открывает карточку:

```
┌────────────────────────────────────────────────────────────┐
│ #47 ⚠ NEEDS APPROVAL · git-push                       [✕] │
│ создано: бэкенд, 2 мин назад · родитель: #42                │
├────────────────────────────────────────────────────────────┤
│                                                             │
│ git push origin main для коммита fd9ecdb                   │
│                                                             │
│ Бэкенд просит подтвердить:                                  │
│                                                             │
│   Команда:     git push origin main                         │
│   Файлы:       pride_dev/webhook.py, pride_dev/bitrix.py    │
│   Коммит:      fd9ecdb «Webhook-приёмник Bitrix24»          │
│   Тесты:       14/14 зелёные                                │
│   Origin:      git@gitlab.com:rdm9x-group/pride-dev.git     │
│                                                             │
│ Что произойдёт после approve:                               │
│ - GitLab получит код                                        │
│ - CI/CD trigger (если настроен)                             │
│                                                             │
│                          [✗ Reject]   [✓ Approve]           │
└────────────────────────────────────────────────────────────┘
```

Дмитрий читает, нажимает одну из кнопок.

### После Approve

1. Дашборд → backend → PATCH /api/tasks/47/approve.
2. Backend меняет статус задачи #47 на `wip`.
3. Добавляется коммент в #47 от «дмитрий»: «approved at 14:32».
4. Subagent (или тимлид) при следующем проходе видит approved → выполняет команду.
5. После выполнения → submit_result в #47 → status `done`.
6. Родительская задача #42 разблокируется → тимлид продолжает работу.

### После Reject

1. Аналогично, но статус → `done` с пометкой `rejected`.
2. Subagent получает уведомление через коммент в родительской задаче.
3. Тимлид решает что делать: переформулировать, эскалировать Дмитрию для уточнения, или закрыть родительскую задачу как cancelled.

### Опциональный третий вариант — «Approve с условием»

Когда Дмитрий хочет approve, но с правкой:
```
Approve с комментарием: «push сделай, но в ветку dev, не main»
```

Это просто approve + комментарий. Subagent читает коммент и адаптирует команду.

## Где это всё хранится в БД

В таблице `tasks` — поля:
- `requires_approval BOOLEAN DEFAULT 0` — флаг что задача в approval-flow.
- `status = "needs_approval"` — текущее состояние.
- `labels` JSON — содержит `"approval"` + конкретный тип `"git-push"` / `"ssh-deploy"` / etc.

В таблице `task_comments` — отметка о решении Дмитрия:
- author=`"дмитрий"`
- text=`"approved at <ts>"` или `"rejected at <ts>: <reason>"`.

## Что в промтах ролей

В каждом промте роли (тимлид, бэкенд, QA) явно прописано:

> **Перед выполнением любой операции из списка «требует approval» — ОБЯЗАТЕЛЬНО создать task со status="needs_approval" и **ждать** решения Дмитрия в комментарии. НЕ выполнять операцию авансом. НЕ обходить через временные скрипты. НЕ договариваться сам с собой.**

Это часть identity роли — она не должна вертеть approval-gates как формальность.

## Тестирование approval-flow

Включить в smoke-тест:
1. Тимлид делегирует бэкенду «закоммить hello.py».
2. Бэкенд делает code+commit, создаёт needs_approval для push.
3. Дмитрий в дашборде нажимает Approve.
4. Бэкенд при следующем проходе делает push.
5. Тимлид закрывает родительскую.

Без этого MVP считать неготовым.

# devboard — малая команда разработки

Малая автономная команда из 3 ролей-ботов (тимлид, бэкенд, qa) на твоём Mac.
Её единственная задача — построить большую систему `dev-department`
(локальный AI-офис). См. [КОНЦЕПТ.md](КОНЦЕПТ.md) для общей картины.

## Что внутри

```
devboard/
  КОНЦЕПТ.md                  — общая идея двухуровневой системы
  детали_дашборда.md          — спека UI
  approval_gates.md           — workflow подтверждений
  ПРОМТ_РЕАЛИЗАТОРУ.md        — задание этой сессии
  .mcp.json                   — конфиг MCP для тимлида
  pytest.ini                  — единый прогон тестов
  README.md                   — этот файл
  data/
    tasks.db                  — SQLite канбан (создаётся автоматически)
    tasks.db.lock             — fcntl-lock файл
    dashboard.log             — лог Flask
    team.log                  — live-лог сессий тимлида
  mcp_server/                 — MCP-сервер devboard-tasks (8 tools)
    devboard_tasks/{db,tools,models,server}.py
    tests/                    — 51 тест
  dashboard/                    — Flask
    app.py
    templates/kanban.html
    static/{style.css,app.js}
    tests/                    — 16 тестов
  roles/                      — системные промты тимлида/бэкенда/qa
  commands/
    devboard-start.sh       — запуск дашборда
    devboard-stop.sh        — остановка всего
    devboard-work.sh        — внутренний: запускает claude как тимлида
                                (вызывается дашбордом из «▶ Запустить команду»)
```

## Возможности

- **Канбан-доска** — задачи проходят TO DO → WIP → NEEDS APPROVAL → REVIEW → DONE. Опасные операции ты подтверждаешь вручную, остальное делают агенты.
- **Лайв-лог** — поток вывода от Claude сессии разбирается в читаемые строки и отдаётся в браузер через SSE.
- **Approval-gate** — `git push`, `ssh`, `systemctl restart` и другие рискованные команды требуют твоего явного подтверждения.
- **Роутер моделей** — выбирает `haiku`, `sonnet` или `opus` по сложности задачи. Токены на routing не тратятся.
- **Вкладка «Настройки»** — язык, режим тимлида и предпочтения модели меняются прямо в дашборде. Руками редактировать `.env` не нужно.
- **Вкладка «Статистика»** — пропускная способность задач, скорость команды и аналитика по ролям из того же SQLite.
- **Двуязычный i18n** — и интерфейс дашборда, и вывод агентов переключаются между RU и EN одним тоглом.
- **Режим plain-language** — тимлид упрощает вывод для нетехнических продакт-менеджеров. Включается в настройках, без редактирования промтов.
- **Мультиролевая команда** — тимлид, бэкенд, QA работают по умолчанию. Архитектор, фронтенд, DevOps, техписатель — подключаются как модули.

## Быстрый старт (рекомендуется)

```bash
git clone https://github.com/your-org/devboard.git
cd devboard
cp .env.example .env
# Открой .env и пропиши ANTHROPIC_API_KEY
docker compose up
# Открыть http://localhost:4999
```

Работает одинаково на **Windows / macOS / Linux**. Требуется [Docker Desktop](https://www.docker.com/products/docker-desktop/).

Данные (`data/tasks.db` и логи) хранятся на хосте в папке `./data` — переживают перезапуск и пересборку контейнера.

После запуска:

1. Жми **«+ Новая задача»** → заполни форму → задача появится в TO DO.
2. Жми **«▶ Запустить команду»** в шапке. Тимлид прочитает канбан, декомпозирует задачу, делегирует subagent'ам. Live-вывод стримится в нижнюю панель.
3. Когда тимлид перевёл задачу в **REVIEW** — открывай карточку, жми **«Принять»** или **«На доработку»**.
4. Если задача в **NEEDS APPROVAL ⚠** — прочитай что собираются сделать, жми **Approve** или **Reject**.

## Ручная установка (без Docker)

> Используй этот путь, если Docker недоступен. Требуется Python ≥3.11 и `uv`.

Зависимости устанавливает `devboard-start.sh` при первом запуске —
руками ничего делать не нужно. Если хочется убедиться заранее:

```bash
cd ~/devboard/mcp_server
uv venv && uv pip install -e ".[dev]"

cd ../dashboard
uv venv && uv pip install -e . && uv pip install pytest
```

## Запуск

```bash
cd ~/devboard
./commands/devboard-start.sh
```

Откроется на **http://127.0.0.1:4999** (порт меняется через
`DEVBOARD_DASHBOARD_PORT=5001 ./commands/devboard-start.sh`).

Язык интерфейса, режим тимлида и выбор модели настраиваются в **вкладке «Настройки»** прямо в дашборде.

В дашборде:
1. Жми **«+ Новая задача»** → заполни форму → задача появится в TO DO.
2. Жми **«▶ Запустить команду»** в шапке. Flask запустит сессию `claude -p`
   с системным промтом тимлида + MCP `devboard-tasks`. Тимлид прочитает канбан,
   декомпозирует задачу, делегирует subagent'ам (бэкенд/qa), соберёт
   результаты, обновит статусы. Live-вывод стримится в нижнюю панель.
3. Когда тимлид перевёл задачу в **REVIEW** — открывай карточку,
   читай результат, жми **«Принять»** (→ done) или **«На доработку»**.
4. Если subagent создал задачу в **NEEDS APPROVAL ⚠** — открой её, прочитай
   что собираются сделать, жми **Approve** или **Reject**.

## Мультикомандный режим (v2.0)

С v2.0 devboard перестал быть «одна команда — один канбан» и стал **платформой AI-отделов**. В одном инстансе живут несколько отделов: `Dev`, `Marketing`, `Design`, `Sales`, `Support`, `Operations` — у каждого свой канбан, свой набор ролей и свой чат. Текущий отдел хранится в `localStorage` и передаётся в каждом запросе через заголовок `X-Department`. Старые вызовы `/api/tasks` и `/api/chat` без заголовка по-прежнему возвращают данные отдела `dev` — это сохраняет совместимость с v1.x.

Новые отделы создаёт **глобальная роль HR** через чат-диалог. HR подбирает один из пяти встроенных шаблонов (`marketing-v1`, `design-v1`, `sales-v1`, `support-v1`, `operations-v1`), кастомизирует его под тебя в 1–5 итераций edit-loop и записывает файлы ролей только после твоего approve. История кастомизаций — в `extras.hr_meta` каждого файла роли и в таблице `hr_sessions`.

Между отделами действует строгий **протокол Lead-to-Lead**. Кросс-задачу может создать только Lead отдела-заказчика (или owner) через `POST /api/departments/<target>/tasks`. Принимающий Lead может либо «взять в очередь», либо counter-propose — кнопки «Decline» нет принципиально. Задачи приоритетов `P1`/`P2` и всё с лейблом `requires_budget` уходят на approve owner'у. Глобальный append-only канал `inter-department` собирает аудит всех cross-task событий.

Архитектурные подробности — в трёх ADR: [ADR-003](docs/adr/0003-departments.md) (модель данных), [ADR-004](docs/adr/0004-hr-role.md) (HR-роль), [ADR-005](docs/adr/0005-inter-department.md) (cross-department workflow).

Обновление с v1.x → v2.0 — автоматическое и идемпотентное. Все существующие задачи, роли и сообщения чата переезжают в `department_id = 'dev'`. Подробности — в **[гайде по миграции](docs/migration-v2.md)**.

## Подача первой боевой задачи

В дашборде создай задачу для тимлида (assignee=`тимлид`, priority=`P1`):

> «Спроектируй MVP большой системы (`dev-department`):
> схема таблиц + webhook-приёмник для внешнего сервиса + один виртуальный
> бот Тимлид + acceptance критерий».

Жми «▶ Запустить команду». Тимлид:
1. Декомпозирует на под-задачи (схема, webhook-стек, бот, acceptance).
2. Делегирует бэкенду код-задачи, QA — acceptance-критерии и тесты.
3. По окончанию переведёт родительскую в REVIEW.

Ты увидишь результат в карточке + сможешь принять/отправить на доработку.

## Approval-gate в действии

Когда бэкенд хочет сделать `git push`, он:
1. Создаёт под-задачу `status=needs_approval, labels=["approval", "git-push"]`
   с описанием команды.
2. Завершает работу с пометкой «жду approval».
3. Ты в дашборде видишь её в колонке **NEEDS APPROVAL ⚠**.
4. Открываешь → жмёшь **Approve** или **Reject**.
5. При следующем запуске тимлида он увидит approve и выполнит команду.

Полный список критичных операций — [approval_gates.md](approval_gates.md).

## Как добавить новую роль (на будущее)

1. Создай `roles/<имя>.md` по образцу `roles/бэкенд.md`.
2. В `mcp_server/devboard_tasks/db.py` добавь роль в `_DEFAULT_ROLES`
   (или вставь в БД вручную через `tools.list_roles` + миграция).
3. В `mcp_server/devboard_tasks/models.py` добавь имя в кортеж `ROLES`.
4. Обнови промт тимлида (`roles/dev/lead.md` §«Кто у тебя в команде»).
5. Прогон тестов: `pytest`.

## Архитектурные заметки

- **Атомарность записи в канбан.** Используется fcntl-lock на `data/tasks.db.lock`
  + SQLite `BEGIN IMMEDIATE`. Покрыт параллельным тестом на 8 writer'ов.
- **MCP-сервер** запускается как stdio-процесс из самой сессии claude через
  `.mcp.json`. Отдельный демон не нужен.
- **Дашборд** работает с тем же SQLite напрямую через `devboard_tasks.tools` —
  не через MCP. Для MVP это проще; в production можно перевести на MCP-клиент.
- **`devboard-work.sh`** запускает `claude --print` (one-shot) с
  `permission-mode=bypassPermissions`. Безопасность держится на promt'ах ролей,
  которые обязаны выводить критичные операции через approval-gate. См. там же
  `approval_gates.md` и `roles/*.md`.
- **Модель.** Тимлид — `opus` (длинный контекст + хорошее планирование).
  Subagent'ов тимлид запускает через Task tool со стандартным
  `subagent_type=general-purpose`.

## Известные ограничения MVP

- Один Mac — один пользователь. Multi-user не поддерживается.
- Подписочный лимит Anthropic — если тимлид + N subagent'ов одновременно
  упрутся в rate-limit, claude вернёт ошибку. Тимлид при следующем запуске
  должен продолжить с того же места (state живёт в SQLite).
- SSE-стрим теряется при перезагрузке страницы — это OK, новый коннект
  подцепится к тому же queue. История сессии — в `data/team.log`.
- Авторизация дашборда отсутствует (localhost, только пользователь).
- Notifications/email/push — не реализованы. Только refresh-polling раз в 3 сек.
